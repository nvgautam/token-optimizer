## Key decisions
- **Root cause, not a config gap**: `ContentRouterConfig.protect_recent_reads_fraction` (`content_router.py:647`, default `0.0` = protect all) gets hardcoded to `0.3` in `server.py:637` whenever `is_token_mode(config.mode)` is true (our proxy runs `--mode token`). No CLI/env override exists for this field today — `HEADROOM_PROTECT_RECENT` (`proxy.py:846`) maps to a different field (`protect_recent_code`).
- Effect: `Read`/`Glob` results older than the most-recent-30%-of-messages window fall through `DEFAULT_EXCLUDE_TOOLS` into compression (`content_router.py:2525`, `:2944`) — this is upstream's *intended* tradeoff for generic coding agents, not a defect, but it's wrong for AgentFlow's precision-critical rereads of small state docs.
- Separately observed: `Bash` output (never in `DEFAULT_EXCLUDE_TOOLS` by design — "ideal compression target" per `config.py:212`) degrades under context-pressure-adaptive compression (`min_ratio_relaxed`/`min_ratio_aggressive`) as the session grows — same `cat design_status.md` command was clean early this session, garbled later. Not a new bug, just the documented adaptive-ratio behavior compounding the Read issue.
- Rejected: patching `.venv/site-packages/headroom` directly — not durable, wiped on reinstall, and doesn't serve the user's actual goal (bundling headroom compression as part of AgentFlow's own download).
- Decided path: git-clone upstream `chopratejas/headroom` (has a Rust core under `crates/` — needed if the fix ever touches Rust, though this particular fix looks Python-only) and patch there, so it's redistributable and diffable against upstream.
- No config-level "compress code, not markdown" knob exists in headroom today (checked `config.py`, `content_router.py` — only tool-name exclusion + byte-size thresholds, no content-type toggle). Confirmed this isn't actually the fix needed anyway — `Read` is already supposed to be fully excluded regardless of file type; fixing the exclusion-aging bug matters more than adding content-type gating.
