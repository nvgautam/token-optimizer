# /drift — Architecture Drift Audit

Audit the current project against its agreed design. Run this before starting any significant new work, or whenever something feels off.

Requires `architecture.md`, `tasks.json`, and `CLAUDE.md` to exist in the project root (generated by `/oracle`).

---

## What to audit

### 1. File size violations
Scan all implementation files (exclude `tests/`, stubs, and config). Flag any file exceeding 250 lines.
```
DRIFT [size] src/auth/middleware.py — 312 lines (ceiling: 250)
```

### 2. Undeclared cross-module imports
Read `architecture.md` Module Boundaries and Shared Interfaces sections. For each source file, check its imports. Flag any import that crosses a module boundary without being declared as a Shared Interface.
```
DRIFT [boundary] src/billing/invoice.py imports from src/auth/models.py — not in Shared Interfaces
```

### 3. File ownership conflicts
Read `tasks.json` `owns` declarations. Flag any file that is owned by more than one task, or any file that exists in the codebase but has no declared owner.
```
DRIFT [ownership] src/payments/processor.py — no declared owner in tasks.json
```

### 4. Test coverage gaps
For every implementation file in `owns` lists, check that a corresponding test file exists. Flag missing test files.
```
DRIFT [tests] src/rca/analyzer.py — no test file found
```

### 5. Constraint drift
Read compliance requirements and secrets handling strategy from `architecture.md` Security Model section. Check:
- No hardcoded strings matching secret patterns (API keys, passwords, tokens) in any source file
- Compliance constraints mentioned in architecture.md are present as test assertions (grep for constraint keywords in test files)
```
DRIFT [security] src/config/settings.py:14 — hardcoded string matches secret pattern
DRIFT [compliance] GDPR declared in architecture.md — no test asserting data deletion found
```

### 6. Undeclared integrations
Read the External Integrations table from `architecture.md`. Scan all source files for HTTP clients, SDK imports, and third-party package usage. Flag any external service call not declared in the integrations table.
```
DRIFT [integration] src/notifications/email.py imports sendgrid — not declared in architecture.md
```

### 7. CLAUDE.md staleness
Compare `CLAUDE.md` module list against `architecture.md` Module Boundaries. Flag any module present in one but not the other.
```
DRIFT [claude.md] Module 'reporting' in architecture.md not listed in CLAUDE.md
```

---

## Output format

Group findings by severity:

```
CRITICAL  (block new work — fix before proceeding)
  [list]

WARNING   (fix soon — will compound if ignored)
  [list]

INFO      (intentional evolution candidate — confirm with user)
  [list]
```

For each INFO item, ask: "Is this intentional? If yes, I will update architecture.md and CLAUDE.md to reflect the change."

If the user confirms a drift is intentional:
1. Update `architecture.md` with the new design decision
2. Update `CLAUDE.md` structure and constraints sections
3. Mark the item resolved

---

## No findings

If the codebase is clean, report:

```
No drift detected.
  Files audited:        N
  Modules checked:      N
  Integrations verified: N
  Constraints verified: N
```
